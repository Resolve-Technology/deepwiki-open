# Self-Correcting Generation + Apply-Review — Design

**Date:** 2026-06-05
**Status:** Approved
**Builds on:** 2026-06-05-per-model-wiki-cache-design.md (versioned caches, Model Review, rag_query)

## Problem

1. First-time wiki generation has no quality gate: whatever the LLM writes per page is
   what gets saved, including hallucinated APIs and wrong claims about the code.
2. Model Reviews (cross-model critiques) are stored and displayed, but their findings are
   dead text — there is no way to feed them back into the wiki content.

## Approved decisions

1. **Self-review:** per-page correction pass, toggleable (default on) — right after a page
   is generated, a fresh request to the *same* LLM reviews it against the repo code (RAG)
   and returns a corrected page. Each websocket request is stateless, so "new session" =
   a separate request carrying no prior context.
2. **Apply-review:** "update the output with review content" = the LLM revises the pages
   (not appending the review as a page).
3. **Revision actor/scope:** the wiki's original generating model applies the fixes; a
   cheap classification call first lists which pages the review affects, only those are
   revised. The confirmation dialog listing affected pages is the user's choose-or-not gate.
4. **Saving:** corrected pages overwrite the same provider/model cache version;
   `generated_at` refreshes server-side.

## Design

### Shared plumbing — `src/utils/wikiRevision.ts` (new)

- `runChatOnce(request)` — promise wrapper over `createChatWebSocket`: accumulates the
  stream, resolves with the text on clean close, rejects on socket error, empty response,
  `Error:`-prefixed backend failure text, or timeout (10 min). Used by both features.
- `buildPageRagQuery(page)` — short retrieval query (title + filePaths) so the correction
  prompts get code context despite exceeding the websocket's 8000-token RAG gate
  (reuses the `rag_query` mechanism from the Model Review feature).
- `buildSelfReviewPrompt(page, repoUrl)` / `buildApplyReviewPrompt(page, review, repoUrl)`
  — both demand either the full corrected page markdown or the literal `NO_CHANGES`,
  same language as the page, no preamble.
- `buildAffectedPagesPrompt(review, pages)` / `parseAffectedPages(response, pages)` —
  classification call returns exact page titles one per line (or `NONE`); parsed by
  exact case-insensitive title match against the known page list (unknown lines ignored).
- `parseRevisedContent(original, response)` — the safety gate: returns the original
  unchanged when the response is `NO_CHANGES`, starts with `Error:`, or is shorter than
  30% of the original (a truncated/refused correction must never destroy a good page).
  Strips ```markdown fences like the generation path does.

### Feature 1 — self-review pass (page.tsx + ModelSelectionModal)

- Toggle: `self_review` URL param (default true) → `isSelfReviewEnabled` state → checkbox
  in the generation modal next to the wiki-type selector (rendered only with
  `showWikiType`, so the Ask modal is unaffected).
- In `generatePageContent`, at the single completion point (after both WS and HTTP
  fallback paths converge and markdown fences are stripped, before `setGeneratedPages`):
  when enabled and content is non-error, set a "Reviewing {title}…" loading message and
  `await runChatOnce(...)` with the self-review prompt + `rag_query`; apply
  `parseRevisedContent`. Any failure logs and keeps the original. The page stays in
  `pagesInProgress` during the correction, so the existing all-pages-done save flow and
  progress UI are untouched. `isSelfReviewEnabled` joins the callback's deps.
- Cache metadata: optional `self_reviewed` boolean on `WikiCacheRequest`/`WikiCacheData`
  (backend passthrough); the "Generated by …" line under page titles appends
  "· self-reviewed" when set; `wikiMeta` state carries it.

### Feature 2 — apply review (WikiReviewModal + page.tsx)

- The modal already holds everything needed (pages, review, the wiki's provider/model as
  `reviewedProvider/Model`, repoInfo, token, language), so it orchestrates:
  1. **Classify**: `runChatOnce` with the affected-pages prompt (wiki's own model, no RAG).
  2. **Confirm**: modal shows the affected page titles with Apply/Cancel — the user gate.
     `NONE`/empty → "review needs no content changes" notice, stop.
  3. **Revise**: sequentially per affected page, `runChatOnce` with the apply-review
     prompt + `rag_query`; `parseRevisedContent` guards each. Progress counter shown.
  4. **Hand off**: `onPagesRevised(updatedPages)` callback delivers only the genuinely
     changed pages; failures keep originals and are counted in a summary line
     ("Revised 3 pages, 1 failed, 1 unchanged").
- Apply button on the live review and each saved review, **enabled only when the
  review's `reviewed_provider`/`reviewed_model` match the currently loaded wiki's**
  (you can't apply a review of model A's wiki onto model B's content).
- page.tsx implements `onPagesRevised`: merges into `generatedPages`, then explicitly
  POSTs `/api/wiki_cache` (the auto-save effect is gated by `cacheLoadedSuccessfully`,
  which is true for a loaded wiki, so an explicit save is required) and refreshes
  `wikiMeta` from the response.

### No backend changes beyond metadata

The websocket chat + `rag_query` + review storage all exist. Only `self_reviewed` is added
to the two cache models (optional, default None — old caches unaffected).

### Cost

Self-review ≈ 2× first-generation tokens (hence the toggle). Apply-review = one small
classification call + one generation-sized call per affected page.

### Testing

- Backend: one appended pytest (self_reviewed save/read roundtrip).
- Frontend: no JS test infra in repo — the pure logic lives in `wikiRevision.ts` with
  deliberate simplicity; gates are `npm run build` plus manual verification (generate
  with self-review on/off; apply a review and confirm only affected pages change and the
  save persists across reload).
